📡 RSS / Atom — 覆盖 80%+ 信源的主力渠道
3 工具 全部免费 ▼

绝大多数信源（官方博客、科技媒体、学术论文、聚合站）都提供标准 RSS/Atom feed，这是最稳定、最高效的采集方式。

① feedparser → ② x-reader → ③ Jina Reader
优先级工具类型适合场景速度输出配置需求风险/限制
★1 feedparser Python 库 标准 RSS/Atom feed 解析，快速获取标题+摘要+链接 ⚡ 极快 标题 + 摘要 + URL + 发布时间 pip install feedparser（已在 requirements.txt） 无风险。仅获取 feed 提供的内容，不含全文
★2 x-reader CLI 工具 feedparser 失败时的降级；也可用于获取 feed 全文 🟡 中 Markdown 格式全文 系统安装：/Library/Frameworks/Python.framework/Versions/3.13/bin/x-reader 依赖 Playwright 渲染，速度较慢
★3 Jina Reader 在线 API 备用降级方案 🟡 中 Markdown 格式 无需配置，直接 curl https://r.jina.ai/{URL} 有速率限制（免费 ~1000次/天），依赖外部服务

适用信源类型

  • 官方博客（OpenAI Blog, Google AI Blog 等）
  • 科技媒体（TechCrunch, The Verge, 36氪等）
  • 学术预印本（arXiv, Hugging Face Papers）
  • 聚合站（Hacker News, Techmeme）
  • 所有标记为 type: "rss" 的信源

采集深度说明

  • title_only：feedparser 直接返回标题+摘要，最快
  • full_text：先 feedparser 拿列表，再逐条用 web 渠道抓全文
  • 部分 feed（如 TechCrunch）自带全文内容
🐦 X / Twitter — AI 公司官方 + 行业领袖动态
3 工具 免费为主 RSSHub 不稳定 ▼

Tier 1 的核心渠道，覆盖所有 AI 公司官方账号和行业领袖。通过 RSSHub 将 Twitter 转为 RSS feed。

① feedparser (RSSHub) → ② xreach CLI → ③ x-reader
优先级工具类型适合场景速度输出配置需求风险/限制
★1 feedparser + RSSHub RSS 代理 快速获取推文列表，标题+链接 ⚡ 快 推文内容 + 时间 + 链接 依赖 rsshub.app 公共实例（或自建） ⚠️ 公共 RSSHub 频繁限流/503，是最大不稳定因素。建议自建 RSSHub 实例
★2 xreach CLI CLI 工具 RSSHub 失败时直接搜索 Twitter 🟡 中 JSON 格式推文列表 系统安装 xreach；调用：xreach search "from:@user" --json -n 10 依赖 Agent Reach 服务，需要 X/Twitter Cookie 登录态
★3 x-reader CLI 工具 最终降级：直接渲染 Twitter 页面 🔴 慢 Markdown 格式 已安装 需要浏览器渲染，速度慢，可能需要登录态

适用信源

  • 22 家 AI 公司官方 Twitter（OpenAI, Anthropic, DeepSeek 等）
  • 11 家科技巨头 Twitter（Apple, Microsoft, NVIDIA 等）
  • 23 位行业领袖 Twitter（Karpathy, LeCun, Altman 等）
  • 所有 rsshub.app/twitter/user/* 格式 URL

关键风险

  • 🔴 RSSHub 公共实例不稳定：频繁 503/限流，是整个系统最大瓶颈
  • 🟡 Twitter 官方 API 已关闭免费层，无法直接用 API
  • 🟡 自建 RSSHub 需要服务器 + Twitter Cookie
  • 建议：优先考虑自建 RSSHub 实例，或使用多个公共实例轮换
🌐 Web 通用网页 — 全文抓取 + 反爬突破，6 级降级
6 工具 4 免费 2 付费可选 ▼

最通用的渠道，用于全文抓取任何网页。主要在 full_text 深度模式下被调用——先从 RSS 拿到文章 URL 列表，再逐篇用 Web 渠道抓全文。

① x-reader → ② Jina Reader → ③ Firecrawl → ④ Crawl4AI → ⑤ Tavily Extract → ⑥ httpx
优先级工具类型适合场景速度输出配置需求风险/限制
★1 x-reader CLI 工具 通用网页全文提取，自带 Readability 算法提取正文 🟡 中 Markdown 全文 已安装。支持微信公众号、小红书、Twitter、Telegram 等特殊平台 部分 JS 重渲染站点可能需要 Playwright 回退（自动）
★2 Jina Reader 在线 API x-reader 失败时的快速降级 ⚡ 快 Markdown 全文 无需配置 免费额度 ~1000次/天，超出后限流
★3 Firecrawl 在线 API 高质量结构化提取，反爬能力强 🟡 中 Markdown + 结构化数据 需要 FIRECRAWL_API_KEY 环境变量 ⚠️ 免费 500 次/月，$19/月 3000 次。生产环境需付费
★4 Crawl4AI Python 库 本地浏览器爬取，支持 JS 渲染 🔴 慢 Markdown 全文 pip install crawl4ai，需 Chromium 本地运行，资源消耗大，但无 API 限制
★5 Tavily Extract 在线 API 反爬站点、需要高级提取（LinkedIn 等） 🟡 中 结构化全文 需要 TAVILY_API_KEY（已配置硬编码默认值） 免费 1000 次/月（搜索+提取共享额度）
★6 httpx Python 库 最终降级：直接 HTTP 请求，无 JS 渲染 ⚡ 极快 原始 HTML pip install httpx（已在 requirements.txt） 无法处理 JS 渲染页面，易被反爬拦截

适用场景

  • 深度分析：RSS 返回标题后，需要抓取完整文章正文
  • 无 RSS 信源：部分网站无 RSS feed，直接爬网页
  • 付费墙突破：配合 BrowserWing 带 Cookie 访问
  • 标记为 type: "web_scrape" 的信源

配置清单

  • x-reader：已安装，无需额外配置
  • Jina Reader：无需配置
  • FIRECRAWL_API_KEY：可选，环境变量
  • TAVILY_API_KEY：已有默认值，可用
  • crawl4ai：可选安装

🔍 搜索补充（Exa / Tavily / Multi-Engine） — 主动搜索发现新信息
3 工具 需 API Key ▼

与被动的 RSS/Twitter 不同，搜索渠道是主动发现新内容的方式。用于专题研究、关键词追踪、补充 RSS 覆盖不到的信息。

① Tavily Search → ② Exa AI (mcporter) → ③ Multi-Search Engine
优先级工具适合场景输出配置风险/限制

适用场景

额度管理

  • Tavily 免费 1000次/月，日均 ~33 次
  • Exa 免费额度约 1000次/月

🖥️ 浏览器自动化 — 最终武器：模拟真实浏览器
2 工具 需本地服务 资源消耗大 ▼

所有其他方式都失败时的最终手段。模拟真实 Chrome 浏览器操作，可绕过几乎所有反爬。适用于需要登录态、JS 重渲染、反 bot 防护的站点。

① nodriver → ② BrowserWing
优先级工具适合场景配置风险
★1 nodriver 无头 Chrome 自动化，绕过 Cloudflare 等 pip install nodriver + Chrome 浏览器 资源消耗大，需管理浏览器进程
★2 BrowserWing HTTP API 驱动的浏览器自动化，支持 Cookie 管理 本地服务 localhost:8082，API Key 认证 需要单独启动 BrowserWing 服务

适用场景

  • 需要登录态的付费内容（如 The Information）
  • Cloudflare 等 WAF 保护的站点
  • 重度 JS 渲染的 SPA 应用
  • 需要交互操作（点击、滚动、填表）的采集

BrowserWing 配置

  • 服务地址：http://localhost:8082
  • Cookie 管理：http://localhost:8082/cookies
  • 认证：X-BrowserWing-Key 头
  • Chrome 路径：/Applications/Google Chrome.app
